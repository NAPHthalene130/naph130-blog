理解 Vue 3 组合式 API 的设计思想
2024-06-15 8 min read 2,400 words tags: vue, typescript

Vue 3 的 Composition API 并非凭空出现，它是对 Vue 2 Options API 在使用中暴露出的痛点的一次系统性回应。理解这些设计背景，才能真正用好组合式 API。

设计背景

在 Vue 2 时代，我们按照 data、methods、computed 等选项来组织组件逻辑。小规模组件清晰直观，但随着组件规模的增长，问题逐渐显现——当一个组件涉及多个数据源和交互时，相关逻辑被迫分散在不同的选项中。

组合式 API 不是要取代 Options API，而是提供了一种更灵活的代码组织方式。
const query = ref('')
const results = ref([])
const loading = ref(false)

watch(query, async (val) => {
  loading.value = true
  results.value = await fetchResults(val)
  loading.value = false
})

核心概念

Composition API 的核心思想是按逻辑关注点组织代码，而非按选项类型。相关的状态、计算属性、方法、监听器可以放在一起，形成内聚的逻辑块。

  • useSearch — 搜索逻辑封装，含防抖和结果缓存
  • usePagination — 分页逻辑封装，支持游标和页码两种模式
  • useTheme — 主题切换封装，自动同步系统偏好
  • useFetch — 数据请求封装，含 loading/error 状态管理

总结

Composition API 带给我们的不仅是语法上的变化，更是思维方式上的转变。它鼓励我们将逻辑视为一等公民，让代码组织从"按类型分"走向"按功能聚"。这种编程模型与 React Hooks 殊途同归，代表了前端框架设计的一个共识方向。
