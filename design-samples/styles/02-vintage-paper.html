理解 Vue 3 組合式 API 的設計思想

二〇二四年六月十五日 · 約八分鐘閱讀 · 兩千四百字

目 錄

設計背景 核心概念 實際應用 總結

Vue 3 的 Composition API 並非憑空出現，它是對 Vue 2 Options API 在使用中暴露出的痛點的一次系統性回應。理解這些設計背景，才能真正用好組合式 API。

設計背景

在 Vue 2 時代，我們按照 data、methods、computed 等選項來組織組件邏輯。這種方式在小規模組件中清晰直觀，但隨著組件規模的增長，問題逐漸顯現。

// Vue 2 Options API — 邏輯被選項割裂
export default {
  data() { return { query: '' } },
  watch: { query: 'search' },
  methods: { search() { /* ... */ } }
}

「組合式 API 不是要取代 Options API，而是提供了一種更靈活的代碼組織方式。」

核心概念

Composition API 的核心思想是按邏輯關注點組織代碼，而非按選項類型。相關的狀態、計算屬性、方法、監聽器可以放在一起。

// Vue 3 — 邏輯內聚在一起
const query = ref('')
watch(query, async (val) => {
  results.value = await fetchResults(val)
})

總結

Composition API 帶給我們的不僅是語法上的變化，更是思維方式上的轉變。它鼓勵我們將邏輯視為一等公民，讓代碼組織從「按類型分」走向「按功能聚」。
